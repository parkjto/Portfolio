# 코드 품질 가이드

이 프로젝트는 유지보수성과 코드 품질을 위해 다음 원칙을 따릅니다.

## 원칙

### 1. DRY (Don't Repeat Yourself)
- 중복 코드를 커스텀 훅과 유틸리티 함수로 분리
- 재사용 가능한 컴포넌트 작성

### 2. 단일 책임 원칙 (SRP)
- 각 함수/컴포넌트는 하나의 책임만 가짐
- 복잡한 로직은 작은 함수로 분리

### 3. 명확한 네이밍
- 변수와 함수명은 의도를 명확히 표현
- 약어 사용 최소화

### 4. 에러 처리
- 에러 바운더리로 예외 상황 처리
- 안전한 데이터 접근 (safeGet 등)

## 코드 구조

### 커스텀 훅 (`src/hooks/`)
재사용 가능한 로직을 훅으로 분리:
- `useSlider`: 슬라이더 기능
- `useFullScreen`: 전체화면 모달

### 유틸리티 (`src/utils/`)
공통으로 사용되는 함수와 상수:
- `constants.js`: 전역 상수
- `helpers.js`: 헬퍼 함수

### 컴포넌트 (`src/components/`)
- 각 컴포넌트는 단일 책임
- Props 타입 검증 (향후 PropTypes 추가 예정)
- 에러 바운더리로 감싸기

## 리팩토링 가이드

### 중복 코드 제거
1. 동일한 로직이 3회 이상 반복되면 함수/훅으로 분리
2. 상수는 `constants.js`에 정의
3. 공통 스타일은 CSS 변수나 공통 클래스 사용

### 함수 분리 기준
- 함수 길이: 50줄 이하 권장
- 복잡도: 순환 복잡도 10 이하
- 매개변수: 3개 이하 권장

### 컴포넌트 분리 기준
- 컴포넌트가 200줄 이상이면 분리 고려
- 재사용 가능한 부분은 별도 컴포넌트로
- Props가 5개 이상이면 객체로 그룹화

## 테스트 작성

### 필수 테스트
- 커스텀 훅: 모든 기능 테스트
- 유틸리티 함수: 엣지 케이스 포함
- 컴포넌트: 주요 사용자 시나리오

### 테스트 커버리지
- 최소 80% 이상 유지
- 중요한 비즈니스 로직은 100% 커버리지

## 코드 리뷰 체크리스트

- [ ] 중복 코드가 없는가?
- [ ] 함수/컴포넌트가 단일 책임을 가지는가?
- [ ] 명확한 네이밍인가?
- [ ] 에러 처리가 적절한가?
- [ ] 테스트가 작성되었는가?
- [ ] 주석이 필요한 복잡한 로직에 주석이 있는가?
- [ ] 접근성(aria-label 등)이 고려되었는가?

## 성능 최적화

- 불필요한 리렌더링 방지 (useMemo, useCallback)
- 이미지 최적화 (lazy loading, 적절한 크기)
- 코드 스플리팅 (필요시)

## 접근성

- 시맨틱 HTML 사용
- 키보드 네비게이션 지원
- ARIA 레이블 제공
- 색상 대비 준수
